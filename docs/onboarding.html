1이 프로젝트가 뭔지

한 줄: Glimi 는 살아있는 멀티 에이전트 하네스다. LLM 호출을 8 레이어로 감싸 "오너가 타이핑을 멈춰도 침묵하지 않는" 에이전트 인구를 만든다.

왜 만들어졌나

LLM 은 본질적으로 request-response. 프롬프트 → 응답. 끝. 혼자서 깨어나지 않고, 후속도 안 하고, 먼저 말 걸지도 않는다. 몇 개를 방에 넣어두면 오너가 타이핑 멈추는 순간 방은 조용해진다.

Glimi 는 LLM 호출 하나를 7 reactive 레이어 로 감싸 응답 품질을 잡고, 그 전체를 proactive supervisor 레이어 로 둘러싼다. Supervisor 는 자체 타이머로 돌며 에이전트의 내면 생각 처럼 nudge 를 주입한다. LLM 이 글을 쓰고, 레이어 1-7 이 캐릭터를 지키고, 레이어 8 이 방을 숨 쉬게 한다.

  • Reactive 는 이미 있는 대화를 다듬는다.
  • Proactive 는 없던 대화를 시작한다.

대부분의 멀티 에이전트 프레임워크는 1번밖에 없다. Glimi 는 2번을 추가했다.

현재 상태 (May 2026)

  • 전환 중 피벗 진행 중: 사업화 → OSS career capital 방향. analysis/pivot_subagent_review.md 참조 (gitignored, 메인테이너만 접근)
  • 리팩터 중 커널 추출 중: 도메인 무관 코어 (src/core/*, src/llm/*) 를 src/glimi/ 로 분리해 pip install glimi 라이브러리 화 작업. 2-3주 예상
  • 활성 Hangout 풀-스택 작동: 현재 코드 그대로 Discord 봇 + 웹 대시보드 + 멀티 커뮤니티 가동 가능 (./run.sh)

2두 개의 레이어 — Glimi = umbrella

Glimi 는 단일 제품이 아니라 우산 브랜드. 그 아래에 라이브러리(Core) 와 flagship 앱(Hangout) 이 있다. LangChain → LangSmith/LangGraph 패턴.

두 레이어 구분이 중요한 이유 (코드 차원)

새 기능을 추가할 때 첫 질문: 이건 Core 냐 Hangout 이냐?

  • Core = 도메인 무관. "리서치 두 명이 짝지어 토픽 분석" 시나리오에서도 똑같이 쓸 수 있나? → YES 면 Core
  • Hangout = 도메인 종속. "유나가 친구 셋 만들기 튜토리얼" 처럼 community-sim 메타가 필요 → Hangout
  • 모호하면 Hangout 쪽에 두고, 패턴이 굳어지면 Core 로 승격

3로컬 셋업 — 30분 안에 동작

맥/리눅스 우선. Windows 는 WSL 권장.

필수 사항

  • Python 3.12+
  • Node.js (Claude Code CLI 가 필요)
  • Claude Code CLI: npm install -g @anthropic-ai/claude-code
  • Anthropic API 또는 Claude Code Max 플랜 (페르소나 응답이 Claude 모델로 동작)
  • Discord 봇 토큰 (Hangout 풀-스택 돌리려면 필요. Core 만 만지면 불필요)

한 번에 가동

git clone https://github.com/jaebinsim/Glimi.git
cd Glimi

./run.sh                    # venv 자동 생성 · 의존성 설치 · 플랫폼 + 대시보드 시작
                            # → http://localhost:8000
                            # → 로그인: admin / rmfflal  또는  test / 0000

기타 명령

./run.sh --port 9000                    # 대시보드 포트 변경
./run.sh --imagegen                     # 로컬 LoRA 초상화 생성 (opt-in, ~6분/장)
./run.sh --legacy <community>           # 레거시 단일 봇 모드 (QA / 디버깅)
./scripts/qa.sh                         # E2E QA runner (tmux 세션: Glimi-QA-Runner)
./scripts/stop.sh                       # graceful shutdown

python -m src.platform.accounts list    # 계정 목록
python -m src.community list            # 커뮤니티 목록

주의: 첫 부팅

첫 실행 시 자동으로 venv 생성하고 requirements.txt 설치. 5-10분 걸릴 수 있음. --imagegen 사용 시 추가로 [imagegen] extra (peft + diffusers) 1회 설치 + LoRA 가중치 (.safetensors 186MB) 가 src/glimi_imagegen/loras/ 에 필요.

처음 동작 확인 체크리스트

  • ./run.sh 가 에러 없이 부팅
  • http://localhost:8000 로그인 화면 보임
  • 커뮤니티 하나 생성 → 봇 부팅 → 디스코드 서버에 유나(매니저) 가 첫 인사
  • 유나에게 DM 보내면 응답 옴
  • 대시보드 그래프 탭에서 노드 표시

4아키텍처 — 8 레이어 하네스

모든 LLM 호출이 통과하는 8 단계. 7개는 reactive (응답이 있을 때만), 1개는 proactive (자체 타이머).

📨 메시지 in
   │
   ├─ 1·프롬프트 조립    [Core]  → src/core/prompts/
   ├─ 2·Tool 프로토콜    [Core]  → src/core/tools/
   ├─ 3·메모리 파이프    [Core]  → src/core/memory.py
   ├─ 4·채널 규율        [App]   → runtime.py _describe_channel
   ├─ 5·Anti-echo Guard  [App]
   │
   🤖 LLM 호출 (Haiku / Sonnet / Opus / 로컬)
   │
   ├─ parse · dispatch · dedup
   ├─ 6·A2A 대화 루프    [Core]  → src/core/conversation.py
   ├─ 7·자가 치유        [Tool]  → src/tools/dev_runner.py
   │
📤 메시지 out                          → 비동기 Haiku → 메모리 추출 → DB

⏱ 8·Supervisors  ⭐ [Core, proactive] → src/supervisors/, src/scenes/*/supervisor.py
   tutorial (15초) · chat (15초) · orchestrator (3분)
   ↓
   nudge = 에이전트 본인의 "내면 생각" 처럼 주입

각 레이어 1줄 요약

#레이어분류역할
1프롬프트 조립Core언어 × agent_type dispatch, provider 별 dialect, scene fragment
2Tool 프로토콜Core<tools><call/></tools> XML 파싱, ToolSpec registry
3메모리 파이프라인CoreL0-L3 + facts + relationship + pinned, async Haiku 추출, budget 주입
4채널 규율App"이 채널에서 누가 듣고 있는지" 명시, role bleed 차단
5Anti-echo / GuardApp작별 핑퐁, 단답 ack 재호출, 60s+95% dedup
6A2A 루프Core에이전트끼리 자율 대화 시드, turn limit, closure 감지
7자가 치유Toolruntime 에러 → Opus subprocess 가 소스 패치 → 재시작
8Supervisors ⭐Core입력 없이 도는 유일한 레이어. 페어 스캐너 · 채널 감시 · 씬 진행

5-layer 메모리 시스템

L0 Raw          conversations 테이블 (영구)
L1 Episodic     N msg → digest (Haiku JSON: summary+facts+rel_delta+emotion)
L2 Chronicle    5 L1 → 단락
L3 Saga         5 L2 → 주/월 narrative
+ agent_facts   (subject, predicate, object) + valid_from/valid_to supersession
+ relationship  스냅샷 + history 변곡점
+ pinned        is_pinned=1 (항상 주입)

주입 budget (턴당 ~800 토큰):
  Pinned 400 · Relationship 200 · Episodic-current 700 · Retrieved 400 · Facts 400
Retrieval scoring:
  0.4·semantic + 0.3·importance + 0.2·recency_decay + 0.1·relational

자세한 내용: docs/memory_system.md

6개발 워크플로우

브랜치 정책 · 커밋 · PR 흐름.

브랜치

브랜치역할직접 push
main안정판. 외부 사용자가 보는 기본 브랜치금지
developworking 브랜치. 모든 일반 작업이 여기로메인테이너만
feat/<name> · fix/<name>기여자 브랜치. develop 으로 PR자유

일반 PR 흐름 (외부 기여자)

1. fork + clone
→
2. git checkout -b feat/foo from develop
→
3. 작업 · 커밋
→
4. ./scripts/qa.sh 회귀 확인
→
5. PR → base: develop

커밋 메시지 규칙

  • 짧게. 1줄 제목 (50자 내외). 본문은 정말 필요한 경우만 1-2줄
  • 접두사: feat:, fix:, docs:, ui:, refactor:, test:
  • 한국어 / 영어 둘 다 OK. 일관성만 유지
  • 금지 AI co-author trailer 절대 추가 X — Co-Authored-By: Claude / Generated with X 류 표식 일체 금지
  • 금지 --no-verify, --no-gpg-sign 우회 사용 X — 훅 실패하면 원인 fix
  • 금지 이모지·🤖 같은 AI 생성 표식

예시

feat: bring_friend 도구 — 친한 페르소나가 자기 친구 데려오는 메커니즘

fix: creator 프롬프트 Korean f-string placeholder NameError — '{소개한_친구}' → '<소개한친구이름>'

docs: Glimi umbrella 리브랜딩 (Core + Hangout) + LICENSE Apache-2.0

PR 머지 정책

  • 모든 PR 은 develop 대상
  • 리뷰 + QA 회귀 통과 시 머지
  • 안정화 사이클에서 메인테이너가 develop → main 으로 fast-forward / merge
  • 외부 기여자는 main 직접 건들지 않음

7코드 규칙 — Do / Don't

대부분 CLAUDE.md 에서 가져옴. 한 번만 어겨도 회귀 만들 수 있는 핵심만.

아키텍처 레이어링

✓ Do

  • 새 기능: "이 로직을 Telegram·웹챗 어댑터에서 재사용 가능한가?" 통과해야 Core 자격
  • 플랫폼 중립 타입만 사용 (str, int, dataclass)
  • Discord 결합 코드는 src/bot/ 또는 향후 apps/hangout/ 안에

✗ Don't

  • src/core/* 에서 import discord — 절대 금지
  • Discord 타입 (Webhook, TextChannel, guild) 이 코어 함수 시그니처에 새는 것
  • src/core/prompts/ 에서 src/bot/ 를 import
  • 예외: src/core/sync.py 는 과도기적으로 discord OK (어댑터 책임 — 추후 이동 예정)

프롬프트 작성

✓ Do

  • 위치: 범용 = src/core/prompts/en/ · scene 전용 = src/scenes/{scene}/
  • 영어 정본. 한국 특화 (ㅇㅇ / 카톡 / 호칭) 은 locale.py helper 로 주입
  • 모델 dialect 차이는 src.core.prompts.model.tool_call_syntax_hint() 같은 helper 호출
  • 구조적으로 다르면 ko/{module}.py override

✗ Don't

  • <tools> / <call> 같은 syntax 하드코딩 X (provider 마다 다름)
  • 레거시 [CMD:...] / [ACTION:...] / [QUERY:...] — 전수 제거됨 (커밋 756f3b6), 절대 사용 X
  • 한국어 f-string 안에 {한글_변수} placeholder — NameError 회귀 주의

전체 가이드: docs/prompt_development.md

사용자 보이는 텍스트 (UX)

절대 금지 메타 용어 노출

사용자(오너) 가 보는 텍스트에서 "에이전트", "멤버", "봇", "AI" 등 메타 용어 절대 X. 시스템 프롬프트에선 "다른 사람은 이름/친구들/사람들" 같은 자연스러운 표현.

<tools> 블록은 mgr-system-log 에만 노출 (대화 채널에 절대 X).

상태 관리 / 캐시

✓ Do

  • 메모리/감정 = system prompt 에 안 넣음. agent_runtime 이 user prompt 에 채널별 동적 주입
  • 그룹채팅: 오너 메시지는 handle_group 에서 1회만 로깅 (generate_response 에 log_user_message=False)
  • 프로필 수정 시: invalidate_cache() + runtime.refresh_agent() 쌍으로 호출
  • 타임스탬프 = UTC-aware ISO (datetime.now(timezone.utc).isoformat() 또는 src.core.timeutil.now_utc_iso())

✗ Don't

  • SQLite CURRENT_TIMESTAMP 직접 사용 — UTC naive 라 클라이언트 로컬 tz 렌더 차이 발생
  • dm- / mgr- 채널 삭제 (보호됨)
  • 루트에 봇 토큰·커뮤니티 DB·런타임 상태 두기 (dev/test_defaults.json 만 예외)

안전성

Sync safety brake

src/core/sync.py 에 catastrophic delete 회귀 방지 brake 가 있음. brake 발동 시 자동 재시도 절대 금지 — 수동 점검 후 진행. 자세한 건 analysis/edge_cases.md 참조 (gitignored).

8기여 진입점 — 무엇부터 만지면 좋나

신규 기여자에게 추천 순서. 난이도 색깔: ● easy · ● medium · ● hard.

시작용 · easy

새 example 추가

현재 비어있는 examples/ 슬롯에 도메인별 데모 (research, dev pair, customer support 등). Core API 학습에도 좋음.

시작용 · easy

문서 / README 개선

오타 fix, 다이어그램 추가, 한·영 미러 동기화. README.md ↔ README.ko.md 일관성.

시작용 · easy

새 Hangout scene

src/scenes/ 패턴 따라 birthday · healing · outing 등. docs/scenes_and_supervisors.md 참조.

중급 · medium

로컬 LLM 백엔드

Ollama / vLLM / llama.cpp 구현. src/llm/base.py 의 추상화 + AVAILABLE_MODELS 슬롯이 이미 준비됨.

중급 · medium

관찰성 대시보드 확장

새 메트릭, 새 시각화 (e.g. 메모리 그래프, supervisor heatmap). src/platform/dashboard/.

중급 · medium

새 도구 (ToolSpec)

src/core/tools/registry.py 에 등록. 도구 design 가이드: 도메인 무관이면 Core, Hangout 종속이면 거기.

고급 · hard

커널 추출 (Phase 1-2)

src/core/* + src/llm/* → src/glimi/* 리팩터. KernelStore ABC 정의, DI 적용. 메인테이너와 사전 동기화 필수.

9테스트 / QA

Glimi 는 LLM 응답이 출력의 일부라 전통적 unit test 비중이 낮다. 대신 시나리오 기반 E2E QA 시스템이 핵심.

실행 방법

./scripts/qa.sh                    # E2E QA runner — tmux 세션 Glimi-QA-Runner
                                   # test_user 봇이 mission 시퀀스 수행 + 결과 평가

판정 기준

  • 각 mission 별로 expected behavior 정의
  • 30분 stuck 시 자동 skip → 다음 mission 으로 20분 우회
  • 로그는 data/qa_runs/

자동화 시스템 자세히: analysis/project_qa_automation.md (gitignored) 혹은 scripts/qa.sh 코드 직접.

전통적 unit test

pytest tests/                      # 빠른 unit / integration
pytest tests/e2e/                  # E2E (시간 걸림)

새 기능 추가 시 가능하면 unit test 추가. Core 쪽 모듈은 도메인 무관이라 unit test 잘 어울림. Hangout/UX 영역은 E2E 위주.
