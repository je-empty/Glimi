1이 프로젝트가 뭔지

한 줄: Glimi 는 살아있는 멀티 에이전트 하네스다. LLM 호출을 8 레이어로 감싸 "오너가 멈춰도 조용해지지 않는" 에이전트 집단을 만든다.

왜 만들어졌나

LLM 은 request-response. 프롬프트 → 응답으로 끝. 스스로 깨어나지도, 후속하지도, 먼저 말하지도 않는다. 몇 개를 두면 오너가 멈추는 즉시 방은 조용하다.

Glimi 는 LLM 호출을 7 reactive 레이어 로 감싸 품질을 잡고, 그 밖을 proactive supervisor 레이어 로 둘러싼다. Supervisor 는 타이머로 돌며 에이전트의 내면 생각처럼 nudge 를 넣는다. LLM 은 글을 쓰고, 레이어 1–7 이 캐릭터를 지키며, 8 이 방을 숨 쉬게 한다.

  • Reactive 는 대화를 다듬는다.
  • Proactive 는 없던 대화를 시작한다.

다른 멀티 에이전트 프레임워크는 1번만 있다. Glimi 는 2번을 추가했다.

현재 상태 (May 2026)

  • 전환 중 피벗 진행 중: 사업화 → OSS career capital 방향. analysis/pivot_subagent_review.md 참조 (gitignored, 메인테이너만 접근)
  • 완료 커널 추출: 도메인 무관 코어 (community/core/*, glimi/llm/*) 를 glimi/ 패키지로 분리 — 채팅 SDK/DB 의존 0, standalone wheel 빌드. pip install glimi 는 PyPI 공개(0.1.0) 예정, 그전엔 소스 설치
  • 활성 Community 풀-스택 작동: 웹 우선 (GLIMI_TRANSPORT=web) — 웹 채팅 + 대시보드 + 멀티 커뮤니티 (./run.sh). 라이브 트랜스포트는 웹 어댑터 (community/adapters/web/)

두 레이어 구분이 중요한 이유 (코드 차원)

새 기능 첫 질문: Core 인가 Community 인가?

  • Core = 도메인 무관. "리서치 두 명이 토픽 분석"에도 그대로 쓰이면 Core
  • Community = 도메인 종속. "유나가 친구 셋 만들기 튜토리얼"처럼 community-sim 메타 필요 → Community
  • 애매하면 Community, 패턴 굳으면 Core 로 승격

3로컬 셋업 — 30분 안에 동작

맥·리눅스 우선, Windows 는 WSL 권장

필수 사항

  • Python 3.12+
  • Node.js (Claude Code CLI 필요)
  • Claude Code CLI: npm install -g @anthropic-ai/claude-code
  • Claude CLI 로그인: 첫 실행 setup 위저드가 Claude 모드 안내 — API 키 입력칸 / 월 상한 없음. 백엔드: Claude · Ollama · Grok CLI · echo

한 번에 가동

git clone https://github.com/jaebinsim/Glimi.git
cd Glimi

./run.sh                    # venv 자동 생성 · 의존성 설치 · 플랫폼 + 대시보드 시작
                            # → http://localhost:8000
                            # → 첫 실행 시 admin 비밀번호 입력 (GLIMI_ADMIN_PASSWORD 로도 지정)

기타 명령

./run.sh --port 9000                    # 대시보드 포트 변경
./run.sh --imagegen                     # 로컬 LoRA 초상화 생성 (opt-in, ~6분/장)
./scripts/qa.sh                         # E2E QA runner (tmux 세션: Glimi-QA-Runner)
./scripts/stop.sh                       # graceful shutdown

python -m community.platform.accounts list   # 계정 목록
python -m community list                      # 커뮤니티 목록

주의: 첫 부팅

첫 실행 시 venv 생성 후 requirements.txt 설치 (5–10분). --imagegen 사용 시 [imagegen] extra( peft + diffusers) 1회 설치, LoRA 가중치(.safetensors 186MB) 는 glimi-community/community/glimi_imagegen/loras/ 필요

처음 동작 확인 체크리스트

  • ./run.sh 가 에러 없이 부팅
  • http://localhost:8000 로그인 화면 표시
  • 커뮤니티 생성 → 부팅 → dm-agent-mgr-001 채널에서 유나(매니저)가 첫 인사
  • 유나에게 메시지 보내면 응답 옴
  • 대시보드 그래프 탭에서 노드 표시

4아키텍처 — 8 레이어 하네스

모든 LLM 호출은 8단계 통과. 7개 reactive, 1개 proactive (타이머)

📨 메시지 in
   │
   ├─ 1·프롬프트 조립    [Core]  → community/core/prompts/
   ├─ 2·Tool 프로토콜    [Core]  → community/core/tools/
   ├─ 3·메모리 파이프    [Core]  → glimi/memory.py
   ├─ 4·채널 규율        [App]   → runtime.py _describe_channel
   ├─ 5·Anti-echo Guard  [App]
   │
   🤖 LLM 호출 (Haiku / Sonnet / Opus / 로컬)
   │
   ├─ parse · dispatch · dedup
   ├─ 6·A2A 대화 루프    [Core]  → glimi/conversation.py
   ├─ 7·자가 치유        [Tool]  → community/tools/dev_runner.py
   │
📤 메시지 out                          → 비동기 Haiku → 메모리 추출 → DB

⏱ 8·Supervisors  ⭐ [Core, proactive] → community/supervisors/, community/scenes/*/supervisor.py
   tutorial (15초) · chat (15초) · orchestrator (3분)
   ↓
   nudge = 에이전트 본인의 "내면 생각" 처럼 주입

각 레이어 1줄 요약

#레이어분류역할
1프롬프트 조립Core언어 × agent_type dispatch, provider 별 dialect, scene fragment
2Tool 프로토콜Core<tools><call/></tools> XML 파싱, ToolSpec registry
3메모리 파이프라인CoreL0–L5 (L1 Working Window 포함) + facts + relationship + pinned, async Haiku 추출, budget 주입
4채널 규율App"이 채널에서 누가 듣고 있는지" 명시, role bleed 차단
5Anti-echo / GuardApp작별 핑퐁, 단답 ack 재호출, 60s+95% dedup
6A2A 루프Core에이전트끼리 자율 대화 시드, turn limit, closure 감지
7자가 치유Toolrequest_dev_fix (기본 OFF) — runtime 에러 → subprocess 가 소스 패치 → 재시작
8Supervisors ⭐Core입력 없이 도는 유일한 레이어. 페어 스캐너 · 채널 감시 · 씬 진행

L0–L5 메모리 시스템

L0 Raw            conversations 테이블 (영구)
L1 Working Window 최근 N msg 슬라이딩 윈도우 (현재 대화 컨텍스트)
L2 Episodic       N msg → digest (Haiku JSON: summary+facts+rel_delta+emotion)
L3 Themes         L2 묶음 → 단락
L4 Narrative      상위 묶음 → 주/월 narrative
L5 Identity       장기 정체성/요약 레이어
+ agent_facts     (subject, predicate, object) + valid_from/valid_to supersession
+ relationship    스냅샷 + history 변곡점
+ pinned          is_pinned=1 (항상 주입)

주입 budget (턴당 ~800 토큰):
  Pinned 400 · Relationship 200 · Episodic-current 700 · Retrieved 400 · Facts 400
Retrieval scoring:
  0.4·semantic + 0.3·importance + 0.2·recency_decay + 0.1·relational

자세한 내용: docs/memory_system.md

6개발 워크플로우

브랜치 정책 · 커밋 · PR 흐름.

브랜치

브랜치역할직접 push
main안정판. 외부 사용자가 보는 기본 브랜치금지
developworking 브랜치. 모든 일반 작업이 여기로메인테이너만
feat/<name> · fix/<name>기여자 브랜치. develop 으로 PR자유
dev-requests/run-{ts}자동 dev 시스템이 사용. 사람이 직접 만들지 말 것자동 전용

일반 PR 흐름 (외부 기여자)

1. fork + clone
→
2. git checkout -b feat/foo from develop
→
3. 작업 · 커밋
→
4. ./scripts/qa.sh 회귀 확인
→
5. PR → base: develop

커밋 메시지 규칙

  • 짧게. 제목 50자 내, 본문 1~2줄만
  • 접두사: feat:, fix:, docs:, ui:, refactor:, test:
  • 한국어 / 영어 둘 다 OK. 일관성만 유지
  • 금지 AI co-author trailer 추가 금지 — Co-Authored-By·Generated with 표식 불가
  • 금지 --no-verify, --no-gpg-sign 사용 금지 — 훅 실패 시 원인 fix
  • 금지 이모지·🤖 같은 AI 생성 표식

예시

feat: bring_friend 도구 — 친한 페르소나가 자기 친구 데려오는 메커니즘

fix: creator 프롬프트 Korean f-string placeholder NameError — '{소개한_친구}' → '<소개한친구이름>'

docs: Glimi umbrella 리브랜딩 (Core + Community) + LICENSE AGPL-3.0

PR 머지 정책

  • 모든 PR 은 develop 대상
  • 리뷰 + QA 회귀 통과 시 머지
  • 안정화 시 메인테이너가 develop → main fast-forward 또는 merge
  • 외부 기여자는 main 직접 건들지 않음

7코드 규칙 — Do / Don't

대부분 CLAUDE.md 기반. 위반 시 회귀 위험 핵심만.

아키텍처 레이어링

✓ Do

  • 새 기능은 "이 로직을 Telegram·웹챗 어댑터에서 재사용 가능한가?" 통과 시 Core 자격
  • 플랫폼 중립 타입만 사용 (str, int, dataclass)
  • 트랜스포트 코드는 community/adapters/web/ (라이브 웹 어댑터) 같은 어댑터 레이어 내

✗ Don't

  • community/core/* 에서 특정 채팅 SDK import 금지 (transport 중립)
  • 채팅 SDK 타입이 코어 함수 시그니처에 새지 않게
  • community/core/prompts/ → 어댑터 레이어 import 금지
  • 새 트랜스포트는 ChannelAdapter Protocol (community/core/channel_adapter.py) + Outbox seam 에 plug-in

프롬프트 작성

✓ Do

  • 위치: 범용 community/core/prompts/en/ · scene별 community/scenes/{scene}/
  • 영문 기준. 한국 특화(locale.py helper로 ㅇㅇ·카톡·호칭 주입)
  • 모델별 dialect 차이는 community.core.prompts.model.tool_call_syntax_hint() 호출
  • 구조 다르면 ko/{module}.py override

✗ Don't

  • <tools>, <call> 등 syntax 하드코딩 금지 (provider마다 다름)
  • 레거시 [CMD:...], [ACTION:...], [QUERY:...] 제거 완료(커밋 756f3b6), 재사용 금지
  • 한국어 f-string의 {한글_변수} placeholder — NameError 주의

가이드: docs/prompt_development.md

사용자 보이는 텍스트 (UX)

절대 금지 메타 용어 노출

사용자(오너)가 보는 텍스트엔 "에이전트", "멤버", "봇", "AI" 등 메타 용어 X. 시스템 프롬프트에선 "이름/친구들/사람들" 등 자연스러운 표현.

<tools> 블록은 채널에 노출 금지. 런타임 도구 로그는 logs/system.log 로 (mgr-system-log 채널 폐지).

상태 관리 / 캐시

✓ Do

  • 메모리·감정은 system prompt 제외. agent_runtime 이 user prompt 에 채널별 동적 주입.
  • 그룹채팅: 오너 메시지는 handle_group 에 1회만 로깅 (generate_response 의 log_user_message=False).
  • 프로필 수정 시 invalidate_cache() 후 runtime.refresh_agent() 호출.
  • 타임스탬프 = UTC-aware ISO (datetime.now(timezone.utc).isoformat() 또는 community.core.timeutil.now_utc_iso()).

✗ Don't

  • SQLite CURRENT_TIMESTAMP 는 UTC naive — 로컬 tz 렌더 차이 발생.
  • dm- / mgr- 채널 삭제 (보호됨)
  • 루트에 봇 토큰·커뮤니티 DB·런타임 상태 두기 (dev/test_defaults.json 예외).

안전성

대량 삭제 주의

과거 Discord↔DB sync 모듈에 catastrophic delete 방지 brake 가 있었으나, Discord 어댑터 제거(2026-06-25)와 함께 해당 모듈도 삭제됨. 현재 라이브 트랜스포트는 웹 채팅 단일 — 외부 소스↔DB 동기화 단계가 없다. 향후 새 어댑터에서 양방향 sync 를 추가한다면 동일한 lookahead+brake 패턴을 다시 적용할 것. 관련 이력은 docs/edge_cases.md 참조.

9테스트 / QA

Glimi 는 LLM 응답 포함 출력 구조로 unit test 비중↓, 시나리오 기반 E2E QA 중심.

실행 방법

./scripts/qa.sh                    # E2E QA runner — tmux 세션 Glimi-QA-Runner
                                   # test_user 봇이 mission 시퀀스 수행 + 결과 평가

판정 기준

  • 각 mission 의 expected behavior 정의
  • 30분 정체 시 자동 skip → 다음 mission 으로 20분 우회
  • 로그는 data/qa_runs/

자동화 세부: analysis/project_qa_automation.md (gitignored) 또는 scripts/qa.sh 참조.

전통적 unit test

pytest tests/                      # 빠른 unit / integration
pytest tests/e2e/                  # E2E (시간 걸림)

새 기능 추가 시 가능하면 unit test 추가. Core 모듈은 도메인 독립이라 unit test 적합. Community/UX는 E2E 중심.
