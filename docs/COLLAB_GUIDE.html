1파트별 역할 분담

사람 이름이 아니라 파트(역할) 단위로 나눈다. 각 파트에 주도 역할이 있고, 파일 단위 규칙은 자유 / 리뷰 필수 / 메인테이너 전용 3구역.

파트 맵

파트범위주도·상태
🧠 코어 src/core/ (runtime·memory·prompts·tools) · src/llm/ · src/db.py 메인테이너 주도. 미완성 — 개발 계속 진행 중. 필요시 누구나 수정 가능, 단 전부 리뷰 필수
🎭 커뮤니티·콘텐츠 도전과제 catalog · 씬 · 유나 FAQ 콘텐츠 오너 주도 — 수요(서브컬처·오시카츠 등) 감각 기반 기획·설계
🖥️ 플랫폼 src/platform/ (FastAPI·대시보드·setup) 메인테이너 주도
🔌 어댑터 src/bot/ (Discord 출구 레이어) 과도기 — 추후 src/adapters/ 분리 예정. 수정 시 decoupling 원칙 준수
⚙️ 인프라·릴리스 run.sh/run.bat · CI/배포 · 시크릿 · 브랜치 관리 · main 릴리스 메인테이너 전용

파트별 진행 중 작업·투두의 단일 소스 = docs/dev_guide.md (현재 스프린트 + 로드맵 섹션). 콘텐츠 파트 백로그는 GitHub 이슈로 관리.

구역 규칙 — 파일 기준

구역경로규칙
자유 src/achievements/catalog/ · src/scenes/<새 씬>/ · docs/yuna_knowledge.md 콘텐츠 오너는 본인 판단으로 작업 → PR. 가드레일 통과 + 씬/도전과제 추가 시 yuna_knowledge.md 같이 갱신. 씬은 4종 세트 (scene + supervisor + handlers + prompts, 레퍼런스 src/scenes/tutorial/)
리뷰 필수 코어 전체 (src/core/ · src/llm/ · src/db.py) · src/achievements/engine.py·definitions.py · src/supervisors/ · src/bot/ · src/platform/ 코어는 잠겨있지 않다 — 개발이 계속 들어가는 영역이라 필요하면 수정 OK. 단 ① 작업 전 의도 1-2줄 공유, ② PR 에 "⚠ core 접점: 파일 — 이유" 명시, ③ 메인테이너 승인 후 머지. 새 도구·프롬프트는 각각 도구 절제 심사 / docs/prompt_development.md 규칙 통과
메인테이너 전용 run.sh/run.bat · CI/배포 설정 · 시크릿·키 · 브랜치 보호 · main 릴리스 변경 필요하면 이슈/메시지로 제안
경계가 애매하면? "이 파일이 콘텐츠인가 시스템인가" 헷갈리면 일단 리뷰 필수로 취급하고 작업 전에 물어보기. 잘못된 구역에서 한참 작업하고 PR 거절당하는 것보다 30초 질문이 싸다.

2작업 워크플로우 5단계

모든 작업은 이 5단계. 로컬 클론 + Claude Code 기준 (바이브코딩 전제).

STEP 1
최신화 + 분기
develop pull 후 새 브랜치 생성
STEP 2
작업
Claude Code 와 작업. CLAUDE.md 가 규칙을 자동 주입
STEP 3
검증
테스트 실행 + 실제 동작 확인 (§7)
STEP 4
커밋 + push
커밋 양식 (§4) 준수, 본인 브랜치에만
STEP 5
PR
base = develop, PR 양식 (§5)
# STEP 1 — 매 작업 시작 시
git checkout develop
git pull origin develop
git checkout -b feat/oshikatsu-scene     # 명명 규칙은 §3

# STEP 4 — 작업 끝나면
git add -A && git commit -m "feat: 오시카츠 씬 추가 — 굿즈 자랑 + 같이 본방사수 페이즈"
git push -u origin feat/oshikatsu-scene

# STEP 5 — PR 생성 (gh CLI 또는 GitHub 웹)
gh pr create --base develop --title "feat: 오시카츠 씬 추가" --body "..."
🚫 GitHub 웹 "Upload files" 절대 금지. 웹 업로드는 (1) 커밋 메시지가 전부 "Add files via upload" 가 되어 히스토리가 죽고, (2) 파일 전체를 통째로 덮어써서 그 사이 다른 사람 변경을 날릴 수 있고 (충돌 감지 없음), (3) 줄바꿈(CRLF)이 뭉개져 diff 가 오염되고, (4) 리뷰 없이 브랜치에 직행한다. 반드시 로컬 클론 → 브랜치 → push → PR. git 이 어렵다면 Claude Code 한테 "이 작업 브랜치 만들어서 커밋하고 PR 올려줘" 라고 하면 전부 해준다 — 그게 이 프로젝트의 표준 방식.

3브랜치 규칙

브랜치역할직접 push
main안정판 — 외부 사용자가 보는 기본 브랜치❌ 절대 금지 (메인테이너만, 안정화 사이클에서)
develop통합 지점 — 모든 PR 의 base❌ 기여자는 PR 로만 (메인테이너 일상 커밋은 브랜치 보호 적용 전까지 과도기 허용)
feat/* fix/* docs/* refactor/*작업 브랜치 — develop 에서 분기✅ 본인 브랜치는 자유
dev-requests/run-*자동 dev 시스템 전용❌ 사람이 만들지 말 것
  • 이름은 영문 소문자 + 하이픈: feat/idol-fan-achievements, fix/scene-phase-skip
  • 브랜치 1개 = 주제 1개. 머지되면 삭제.
  • 오래 끄는 브랜치 금지 — 3일 넘게 develop 과 벌어지면 중간에 develop 을 머지해서 따라잡기.

4커밋 양식

1줄 제목 = 타입: 요약. 필요할 때만 본문 1-2줄. 한국어 OK.

타입용도
feat새 기능·새 콘텐츠 (씬, 도전과제 등)
fix버그 수정
styleUI·문구·디자인 (동작 변화 없음)
docs문서만 변경
refactor동작 동일, 구조 개선

✅ 좋은 예 (실제 히스토리에서)

  • feat: 오시카츠 도전과제 3종 추가 (최애 자랑·굿즈·본방사수)
  • fix: 씬 phase 전환 시 supervisor 미해제로 중복 발화
  • docs: yuna_knowledge 에 신규 씬 FAQ 추가

❌ 나쁜 예

  • Add files via upload ← 무슨 작업인지 0 정보
  • update / 수정 / wip
  • 장황한 본문 + 변경 파일 나열 (diff 가 이미 보여줌)
  • Co-Authored-By: Claude ... ← AI co-author trailer 절대 금지
  • 커밋 1개 = 논리적 변경 1개. "씬 추가 + 무관한 버그픽스" 를 한 커밋에 섞지 않기.
  • --no-verify 금지 — pre-commit 훅이 실패하면 원인을 고친다.
  • AI 생성 표식 (🤖, "Generated with Claude Code") 금지 — author 는 본인 git config.

5PR 양식 + 크기 규칙

base 는 항상 develop. 제목은 커밋 양식과 동일 (타입: 요약).

PR 본문 템플릿

## 무엇
오시카츠 씬 추가 — 최애 자랑 → 같이 본방사수 → 후기 수다 3 페이즈.

## 왜 (수요 근거)
타깃 B 에서 오시카츠는 일상 대화 빈도 최상위 주제.
기존 씬엔 '덕질 공유' 류 감정 접점이 없음.

## 확인한 것
- [ ] tutorial 씬 패턴 (4파일 세트) 그대로 따름
- [ ] ./scripts/qa.sh 통과 / 실제 대화로 phase 전환 확인
- [ ] docs/yuna_knowledge.md 갱신
- [ ] 가드레일 (§6) 통과 — 메타 용어 없음, 해결형 드라마

## 리뷰 포인트 (선택)
supervisor 의 phase 전환 타이밍 판단이 맞는지 봐줬으면 함.

크기 규칙

  • PR 1개 = 주제 1개. "씬 1개 추가" / "도전과제 묶음 1세트" 단위. 여러 주제면 PR 분리.
  • 변경 ~500줄 이하 권장. 넘으면 리뷰 품질이 급락 — 콘텐츠 대량 추가는 세트 단위로 쪼개기.
  • 리뷰 필수 구역을 건드린 PR 은 본문에 "⚠ core 접점: 파일명 — 이유" 명시.
  • 머지는 메인테이너가. 셀프 머지 금지.
리뷰 SLA. 메인테이너는 PR 에 48시간 내 첫 반응 (승인/코멘트/일정 공유). 급한 건 디스코드/메신저로 핑.

6콘텐츠 가드레일 — 커뮤니티 작업 시 필수 통과

수요 감각으로 콘텐츠를 미는 건 환영. 단, 아래는 프로젝트 설계 락인이라 수요가 있어도 안 됨. 전체 맥락은 docs/dev_guide.md 의 "설계 락인 6가지".

체크기준
🎯 타깃20대 초반 여성 감성 유저 (장기관계·힐링·케어). 10대 서브컬처 (이세계·학원 배틀·하드 판타지) 금지. 오시카츠·일상 덕질은 타깃과 잘 맞음 — OK
💚 감정 접점"이 콘텐츠가 친구의 감정 delta 를 남기는가?" NO 면 우선순위 낮춤
🕊️ 드라마갈등은 발생 → 중재 → 화해. 파괴적 엔딩·배신·폭력 금지
🔇 메타 용어사용자 노출 텍스트에 "AI / 봇 / 에이전트 / 멤버" 금지 — 친구·사람·이름으로
🧰 도구 절제새 <tools> 도구는 "기존 도구로 대체 불가 + 감정 접점" 둘 다 통과해야 추가 (리뷰 필수 구역)
🔌 플랫폼 중립코어 로직에 Discord 타입 누출 금지 — "이거 텔레그램·웹챗에서도 재사용 가능한가?" NO 면 잘못된 레이어
🧬 코어 중립코어(src/core/·src/llm/)에 특정 커뮤니티 콘텐츠 하드코딩 금지 — 캐릭터명·실존 아티스트/IP·특정 언어 문구는 커뮤니티 데이터 파일/설정에서 로드. "내 커뮤니티에서 잘 돌게" 만드는 수정은 환영하되, 그 데이터는 communities/ 쪽 레이어로
📖 문서 동기화씬·도전과제 추가 시 docs/yuna_knowledge.md 같이 갱신 (유나가 사용자에게 설명하는 소스)

7머지 전 품질 체크

PR 올리기 전에 돌리는 것. Claude Code 한테 "머지 전 체크 돌려줘" 라고 하면 됨. 전체 테스트 절차 (E2E QA 셋업 + QA 페르소나에 자기 자신 넣기) 는 docs/testing.md.

# 커뮤니티 격리 (global state 만졌으면 필수)
python -m tests.unit.test_community_isolation

# E2E QA (씬·튜토리얼·봇 플로우 변경 시)
./scripts/qa.sh

# 콘텐츠 변경은 실제 대화로 확인 — 테스트 통과 ≠ 체감 품질
# (씬이면 phase 전환 전 구간을 실제로 플레이해보고 스크린샷을 PR 에 첨부)
  • 프롬프트·콘텐츠 변경은 실제 대화 1회 이상 돌려보고 어색한 발화 없는지 확인.
  • 프로필 수정 코드를 건드렸다면 invalidate_cache + runtime.refresh_agent 호출 확인.

8바이브코딩 팁 — Claude Code 와 일하는 법

이 repo 는 Claude Code 친화적으로 세팅돼 있다. 로컬 클론에서 열기만 하면 규칙이 자동 적용됨.

  • CLAUDE.md 가 자동 로드된다 — 브랜치 전략·커밋 규칙·금지사항을 Claude 가 이미 알고 시작. 웹 업로드로는 이 보호막이 전부 무효가 된다 (금지 이유 중 하나).
  • 작업 시작 멘트 예시: "develop 에서 feat/xxx 브랜치 따고, 오시카츠 도전과제 3종 추가해줘. tutorial 씬 패턴 참고." — 브랜치 생성부터 시키면 develop 직커밋 사고가 없다.
  • 마무리 멘트 예시: "머지 전 체크 돌리고, 커밋 + push + develop 베이스로 PR 올려줘."
  • 씬·도전과제 만들 때: 레퍼런스(src/scenes/tutorial/, src/achievements/catalog/oshikatsu.py)를 프롬프트에 명시하면 패턴이 일관돼서 리뷰가 빨라진다.
  • 모르는 건 Claude 한테 먼저: "이 변경이 리뷰 필수 구역 건드려?" / "가드레일 위반 없어?" 같은 셀프 심사도 시킬 수 있다.
  • Claude 가 커밋 메시지에 Co-Authored-By 를 넣으려 하면 거부 — CLAUDE.md 에 금지로 박혀 있어 보통 안 넣지만, 한 번 더 확인.
첫 PR 추천 코스. 작은 것부터: 도전과제 1개 추가 (catalog 파일 1개 + yuna_knowledge 1줄) → PR 플로우 한 바퀴 경험 → 그 다음 씬 같은 큰 단위로. 첫 PR 은 플로우 연습이 목적이라 작을수록 좋다.
