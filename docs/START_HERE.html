1이 프로젝트가 뭔지

한 줄: Glimi 는 살아있는 멀티 에이전트 하네스다. LLM 호출을 8 레이어로 감싸 "오너가 멈춰도 침묵하지 않는" 에이전트 인구를 만든다.

왜 만들어졌나

LLM 은 본질적으로 request-response. 프롬프트 → 응답. 끝. 혼자 깨어나지도, 후속·선제 대화도 없다. 몇 개를 방에 넣어도 오너가 멈추면 방은 조용하다.

Glimi 는 LLM 호출을 7 reactive 레이어 로 감싸 품질을 잡고, 위에 proactive supervisor 레이어 를 둔다. Supervisor 는 타이머로 돌며 에이전트 내면 생각처럼 nudge 를 넣는다.

  • Reactive 는 기존 대화를 다듬는다.
  • Proactive 는 없던 대화를 시작한다.

대부분 프레임워크는 1번뿐이지만, Glimi 는 2번을 둔다.

30초 데모 시나리오 (Community)

14:02 — 너가 잠든 사이, OrchestratorSupervisor 가 3분 타이머에 깨어남
        Haiku judge: "A 와 B 는 1.2h idle, 친밀도 30. 페어 후보."
        → internal-dm-A-B 채널 자동 개설, context="가볍게 근황 토크"

14:03 — A 가 internal-dm-A-B 에서 먼저 말 건다 (오너 모름)
        A: "야 B, 너 요즘 뭐하고 지내?"

14:30 — 너가 dm-B 에 들어가 "뭐해?" 물음
        B: "업무 마감중이야, 방금 A 랑도 얘기했어 ㅋㅋ"
        (방금 대화가 B 메모리에 L1 summary 로 들어가 있음. 친밀도 31.)

Glimi 의 핵심 — 네가 비워도 움직인다.

이건 Core 냐 Community 이냐?

  • Core = 도메인 무관. "리서치 두 명이 토픽 분석" 시나리오에도 같다면 → Core
  • Community = 도메인 종속. "유나가 친구 셋 만들기 튜토리얼" 같은 메타가 필요 → Community
  • 모호하면 Community 에 두고, 패턴이 확정되면 Core 로 승격

6Claude Code 로 개발

프로젝트는 Claude Code 친화적으로 설계됨. 클론 → Claude Code 실행 → 컨텍스트 자동 로드.

자동 로드되는 컨텍스트

이 repo 에서 Claude Code 실행 시 다음이 자동 주입됨:

  • CLAUDE.md (루트) — 아키텍처 원칙, 규칙, 브랜치 전략
  • 각 subdirectory 에 CLAUDE.md 가 있으면 함께 로드 (예: community/glimi_imagegen/CLAUDE.md)

따로 설명 안 해도 Claude 가 "Discord = 어댑터", "AI co-author trailer 금지", "타임스탬프 UTC-aware" 규칙을 알고 시작.

Claude Code 로 작업하는 워크플로우

  1. repo 클론 + Claude Code 실행 
    git clone https://github.com/jaebinsim/Glimi.git
cd Glimi
claude                                  # Claude Code CLI 실행
  2. 첫 명령으로 컨텍스트 확인 
    이 프로젝트가 뭐고 무엇부터 하면 좋을지 START_HERE.html 기반으로 정리해줘

    → Claude 가 CLAUDE.md + 이 HTML 을 요약해 방향 sync.

  3. 새 task 마다 새 브랜치 
    develop 에서 feat/ollama-backend 브랜치 새로 파줘. 그리고 task 시작

    → Claude 가 자동으로 git checkout -b feat/ollama-backend

  4. 여러 단계 작업이면 TodoWrite / TaskCreate 요청 
    이 작업 단계별로 todo 만들어서 진행

    → 진행 상황이 사용자에게 보이고 컨텍스트 복구가 쉬움.

  5. 커밋은 명시적으로 요청

    Claude Code 는 자동 커밋 안 함. "지금 커밋해" 라고 해야 커밋. 메시지는 1줄.

  6. PR 도 명시적으로 
    이 브랜치 develop 에 PR 올려줘

Claude Code 에게 절대 시키면 안 되는 것

✓ Do

  • "develop 에서 새 브랜치 파줘" 명시
  • "이 task 완료 후 커밋해" 명시
  • 여러 단계면 "todo 만들어서" 요청
  • 막히면 "왜 이게 안 되는지 분석해" 부터

✗ Don't

  • "main 에 바로 push" — 금지
  • "커밋 메시지에 Claude 표시" — 금지
  • "--no-verify 로 우회" — 훅 오류는 fix
  • "한 번에 다 해" — 대량 미검증 변경 위험

💡 팁 — CLAUDE.md 가 알아서 막아준다

repo 의 CLAUDE.md 에 "AI co-author 금지", "main 직접 작업 금지", "--no-verify 금지" 규칙이 있어 Claude Code 가 인지 후 시작. 매번 알릴 필요 없음.

7브랜치 전략 — main + develop 만

영구 브랜치 2개. feature/fix 는 임시 분기 → develop 으로 PR.

main ← develop ← feat/<name> 새 작업 분기 / PR base = develop / 메인테이너가 안정화 후 develop → main 머지
브랜치역할누가 push 하나
main 안정판. 외부 사용자가 클론하면 기본으로 받는 브랜치 메인테이너 전용 · 직접 push 금지
develop working 브랜치. 모든 일반 작업의 통합 지점 메인테이너 / PR 머지로만
feat/<name> 기능 개발 한시적 브랜치 기여자 자유 · PR base = develop
fix/<name> 버그 fix 한시적 브랜치 기여자 자유 · PR base = develop
docs/<name> 문서 갱신 한시적 브랜치 기여자 자유 · PR base = develop
refactor/<name> 리팩터링 한시적 브랜치 기여자 자유 · PR base = develop

금지 사항

  • 금지 main 에 직접 커밋 / push
  • 금지 main 으로 PR (모든 PR 은 develop 대상)
  • 금지 임시 브랜치 장기 존치 (머지 후 즉시 삭제)
  • 금지 claude/* 브랜치 (과거 잔재, 2026-05-17 정리됨)

8워크플로우 — 클론에서 PR 까지

외부 기여자 표준 시퀀스.

  1. Fork + Clone 
    git clone https://github.com/<your>/Glimi.git
cd Glimi
git remote add upstream https://github.com/jaebinsim/Glimi.git
  2. develop 최신화 
    git checkout develop
git pull upstream develop
  3. feature 브랜치 분기 
    git checkout -b feat/ollama-backend
  4. 작업 + 로컬 테스트 
    # 코드 변경 작업
./scripts/qa.sh                # E2E QA 회귀 (시간 걸림)
pytest tests/                  # unit test
  5. 커밋 (짧은 메시지) 
    git add <변경 파일>
git commit -m "feat: ollama backend — Gemma 4 / Qwen 3.5 비교 추가"
  6. push + PR 생성 (base: develop) 
    git push origin feat/ollama-backend
# GitHub 에서 PR 생성, base branch = develop
  7. 리뷰 + 머지: 메인테이너가 리뷰 후 develop 에 squash 또는 merge.

커밋 메시지 규칙

  • 1줄 제목. 50자 내. 본문은 필요 시 1-2줄
  • 접두사: feat:, fix:, docs:, ui:, refactor:, test:
  • 한국어 / 영어 둘 다 OK. 일관성만 유지
  • 금지 Co-Authored-By: Claude 등 AI 공동작성자 표기
  • 금지 "Generated with Claude Code", 🤖 등 AI 생성표
  • 금지 --no-verify / --no-gpg-sign 우회

9코드 규칙 — 한번에 회귀 만들기 좋은 것들

CLAUDE.md 출처. 초보가 가장 되돌리기 쉬운 항목.

아키텍처 레이어링

✓ Do

  • 새 기능은 "Telegram·웹챗 어댑터 재사용 가능?" 통과해야 Core 자격
  • 플랫폼 중립 타입만 사용 (str, int, dataclass)
  • Discord 코드는 community/bot/ (또는 apps/community/)에

✗ Don't

  • community/core/* 에서 import discord 금지
  • Discord 타입이 코어 함수 시그니처로 새는 것
  • community/core/prompts/ 가 community/bot/ import
  • 예외: community/core/sync.py 만 임시 허용

프롬프트 작성

✓ Do

  • 위치: 범용 community/core/prompts/en/ · scene 전용 community/scenes/{scene}/
  • 영어 기준. 한국 특화 (ㅇㅇ·카톡) 는 locale.py helper
  • 모델 dialect → model.tool_call_syntax_hint() helper

✗ Don't

  • <tools> syntax 하드코딩 (provider별 상이)
  • 레거시 [CMD:...] / [ACTION:...] / [QUERY:...] 전부 제거
  • 한국어 f-string 내 {한글_변수} placeholder — NameError 주의

사용자 보이는 텍스트 (UX)

금지 메타 용어 노출

사용자 노출 텍스트에 "에이전트"·"멤버"·"봇"·"AI" 금지. 시스템 프롬프트에선 자연스러운 이름·친구 표현만. <tools> 블록은 대화 채널 비노출 — 도구 로그는 logs/system.log.

상태 / 캐시

  • 메모리·감정 system prompt에 넣지 않음 — agent_runtime 이 user prompt에 채널별 주입
  • 프로필 수정 후 invalidate_cache() + runtime.refresh_agent() 호출
  • 타임스탬프 = UTC-aware ISO (src.core.timeutil.now_utc_iso()) — SQLite CURRENT_TIMESTAMP 금지
  • dm- 채널 삭제 금지 (dm-agent-mgr-001 포함, 보호)
  • 루트엔 봇 토큰·커뮤니티 DB·런타임 상태 두지 않음 (dev/test_defaults.json 예외)
